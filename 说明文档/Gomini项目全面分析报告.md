## 二、最重要发现：存在 3 套并行的生肖预测系统

这是当前项目最核心的问题。项目中有三套功能重叠的生肖预测引擎：

### 2.1 三套引擎对比

| 引擎 | 文件 | 级别 | 是否合并到 Business |
|------|------|------|---------------------|
| **Gemini** | `business-gemini.js` + `view-gemini.js` | 最新·主力 | ✅ 已合并 |
| **ZodiacPredict** | `business-zodiac-predict.js` + `view-zodiac-predict.js` | V3.3 旧版 | ❌ 未合并 |
| **ZodiacTiers** | `business-zodiac-tiers.js` | 早期遗存 | ❌ 未合并 |

### 2.2 问题分析

```
❌ 问题1：算法重复
  business-zodiac-predict.js 和 business-gemini.js 都实现了三窗口分池、五大行情判定。
  两套代码逻辑相似，但 Gemini 版本更完整（含 V3.2+V3.3 规则）。

❌ 问题2：数据源不一致
  - Gemini：通过 StateManager._state 直接访问
  - ZodiacPredict：calc(historyData) 接受参数传入
  - 可能导致两套引擎展示不同结果

❌ 问题3：视图重复
  - view-zodiac-predict.js 有独立渲染方法
  - view-gemini.js 有另一套渲染方法
  - 在 MeView 中只调用了 Gemini 的渲染

❌ 问题4：BusinessZodiacTiers 有硬编码 CONFIG
  - 违反了项目宪法"不能修改 CONFIG/常量"
  - 且与实际 config.js 中的配置可能冲突
```

### 2.3 **强烈建议**

> **合并三套为 1 套，以 Gemini 为唯一主引擎。**
>
> 1. 将 `business-zodiac-predict.js` 中的独特功能（如有）移植到 `business-gemini.js`
> 2. 废弃 `business-zodiac-predict.js` 和 `business-zodiac-tiers.js`
> 3. `view-zodiac-predict.js` 改为调用 Gemini 的统一接口
> 4. `business.js` 中只保留 `BusinessGemini`

## 四、数据流分析

### 4.1 当前数据流路径

```
StateManager._state.analysis.historyData  ← 唯一数据源
    │
    ├── BusinessHighChase.calc()      → view-me.js (追号标签)
    ├── ProbabilityView.buildHtml()   → view-me.js (概率学标签)
    ├── BusinessGemini.calc()         → view-gemini.js → view-me.js (Gemini标签)
    └── BusinessZodiacPredict.calc()  → view-zodiac-predict.js (独立渲染)
```

### 4.2 风险点

| 风险 | 说明 |
|------|------|
| 状态访问不一致 | `BusinessGemini._getValidHistory()` 直接读 `StateManager._state`，绕过了 `getState()` 的深拷贝保护 |
| 缓存过期 | `ViewGemini` 60s 缓存可能导致数据与最新状态不同步 |
| 缺乏数据校验 | 没有对 historyData 的完整性检查（如期号连续性、数据格式） |

## 七、性能评估

### 7.1 优化前后对比

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| calc() 遍历历史次数 | ~11次独立遍历 | **1次主遍历 + 1次辅助** |
| 数据结构查找 | 对象属性 O(1) 但多次 | **数组索引 O(1) 且更少** |
| 视图渲染频率 | 每次调用都重新计算 | **60s 缓存 + for-loop/join** |
| 状态更新 | 每次深拷贝 | **直接引用 + 批量防抖** |

### 7.2 进一步优化建议

| 方向 | 预期收益 |
|------|----------|
| Web Worker 后台计算 | 避免 calc() 阻塞 UI 线程 |
| 增量更新替代全量重算 | 新一期数据到达时只更新差值 |
| Lazy DOM 缓存 | `DOM.js` 按需获取而非全量初始化 |
