# Gomini 项目开发方案总纲

> 版本：v2.0 | 更新日期：2026-05-05 | 适用范围：全部开发、修改、新增代码

---

## 第一篇：项目架构与开发规范

### 一、系统概述

Gomini 是一个基于历史开奖数据的多维度智能分析预测系统，集成 **概率学分析**、**高频追号策略**、**生肖预测（Gemini）** 三大核心引擎。

### 二、分层架构（单向调用，严禁反向）

```
平台层（platform/web/）
    ↓ DOM、渲染、Toast、弹窗（仅此处允许DOM操作）
    ↑
核心层（core/）
    ↓ config.js、utils.js、state.js、storage.js
    ↑
业务层（business/）
    ↓ business-high-chase.js（追号策略）
    ↓ business-probability-history.js（概率学）
    ↓ business-gemini.js（生肖预测）
    ↓ business.js（统一合并对象）
    ↑
视图层（views/）
    ↓ view-me.js、view-probability.js、view-gemini.js
    ↑
事件层（event.js）
    ↓ 统一 data-action 事件委托
    ↑
入口层（app.js）
    ↓ 初始化、路由、定时器
```

### 三、架构铁律（绝对不可违反）

| 规则 | 说明 |
|------|------|
| 只能新增，禁止破坏 | 只能新增、追加、扩展、优化代码，禁止删除、覆盖、重写原有功能 |
| 单向依赖 | 只能上层调用下层，严禁反向依赖或循环依赖 |
| 单一职责 | 每个文件只承担单一职责，禁止跨职责混写 |
| 禁止修改 index.html 已有 DOM | 已有结构、id、class 一律不能改动 |
| 禁止修改 CONFIG、常量 | 配置、映射表、基础参数不能变更 |
| 禁止覆盖正常运行函数 | 所有功能只能追加，不能替换 |

### 四、文件职责边界

#### business/ 业务层（绝对禁止）
- ❌ `document.getElementById`、`querySelector`
- ❌ `innerHTML`、`appendChild`、DOM 操作
- ❌ `classList`、`style`、样式修改
- ❌ HTML 渲染、模板生成
- ❌ 事件绑定、Toast、弹窗

#### views/ 视图层（只允许）
- ✅ 渲染 DOM、更新界面
- ✅ 调用业务层拿数据
- ❌ 禁止写业务计算、统计算法

#### event.js 事件层
- ✅ 统一 `data-action` 事件委托
- ❌ 禁止任何渲染代码

#### app.js 入口层
- ✅ 初始化、路由、定时器
- ❌ 禁止任何业务逻辑

### 五、新增功能开发流程

1. 判定是展示 → 写 view
2. 判定是计算 → 写 business
3. 视图只调用业务，业务不依赖视图
4. 不越层、不越权、不乱加

### 六、每次修改必须自检

- [ ] 是否把视图逻辑写到 business 了？
- [ ] 是否在 business 里操作了 DOM？
- [ ] 渲染/排序/事件是否都在 view？
- [ ] 是否没有越界、没有违规？

## 第二篇：核心引擎一 · 概率学分析

### 一、功能定位

通过历史开奖数据计算各生肖的出现频次、遗漏期数，生成概率统计表格，辅助用户判断冷热走势。

### 二、核心模块

| 模块 | 功能 |
|------|------|
| `ProbabilityHistoryView._calcStats()` | 统计各生肖在近5/10/20期的频次与遗漏 |
| `ProbabilityHistoryView._renderTable()` | 渲染频次排名表格 |
| `ProbabilityHistoryView._renderHistory()` | 渲染历史开奖详情 |
| `ProbabilityView.buildHtml()` | 对外统一入口 |

### 三、数据流程

```
StateManager._state.analysis.historyData
    ↓
_extractAllAttrs → DataQuery.getSpecial
    ↓
统计频次、遗漏、五行、波色分布
    ↓

```

### 四、开发规范

- **数据计算**：放在 `business-probability-history.js`（禁止DOM）
- **页面渲染**：放在 `view-probability.js`（禁止计算）

## 第三篇：核心引擎二 · 高频追号策略（动态增强版）

### 一、功能定位

基于动态市场行情判断，自适应调整统计期数、高频阈值、追号期数，输出推荐生肖并提供信心评分和操作建议。

### 二、版本演进

| 版本 | 特性 |
|------|------|
| v1.0 | 固定27期、固定阈值≥3、固定追号3期 |
| v2.0 | 动态参数切换 + 连错风控 |
| v2.1 | 增加信心评分 + 操作建议 |

### 三、核心算法

#### 3.1 行情判断

- **变化率检测**：相邻期数生肖不同的比例 ≥40% 为热
- **种类数检测**：最近10期种类 ≤6 为热
- **满足2条以上** → 热市，否则冷市

#### 3.2 动态参数

| 参数 | 热市 | 冷市 |
|------|------|------|
| 统计期数 | 25期 | 30期 |
| 高频阈值 | ≥3 | ≥2 |
| 追号期数 | 3期 | 2期 |

#### 3.3 追中即停

- 追号期内中奖立即停止
- 3期全错后重置
- 风控暂停：连续错2组 → 观察3期 → 各不相同则恢复

#### 3.4 信心评分

```
评分 = 行情权重(40%) + 高频数量权重(30%) + 命中率权重(30%)
≥80分 → 建议出手
60-79分 → 谨慎出手
<60分 → 建议观望
```

### 四、核心模块清单

| 模块 | 文件 | 职责 |
|------|------|------|
| 行情判断 | `business-high-chase.js` | 热市/冷市判定 |
| 统计排序 | `business-high-chase.js` | 频次统计、近期性排序 |
| 推荐算法 | `business-high-chase.js` | 高频选取、补足推荐 |
| 信心评分 | `business-high-chase.js` | 评分计算、操作建议 |
| 风控控制 | `business-high-chase.js` | 连续错组检测、暂停恢复 |
| 追号控制 | `business-high-chase.js` | 追中即停、剩余期数管理 |

### 五、开发规范

- **所有算法逻辑**：统一写在 `business-high-chase.js`
- **页面渲染**：统一写在视图层对应文件
- **业务层禁止**：任何 DOM、渲染、样式操作

## 第四篇：核心引擎三 · 生肖预测（Gemini）

### 一、功能定位

三窗口交叉验证 + 生肖分池打分 + 五大行情判定 + 12条智能优化规则 + 多维度热度匹配，输出精选3肖。

### 二、前置依赖

项目已内置完整【号码→生肖/五行/波色/尾数/头数/区间】映射关系：

| 接口 | 功能 |
|------|------|
| `DataQuery.getSpecial(item)` | 获取期号特码维度（生肖、五行、波色、尾数、头数等） |
| `DataQuery.getWuxing(num)` | 获取号码五行 |
| `DataQuery.getColorName(num)` | 获取号码波色中文名 |
| `DataQuery.getNumAttrs(num)` | 获取号码全维度属性 |
| `CONFIG.ANALYSIS.ZODIAC_ALL` | 12生肖完整列表 |
| `CONFIG.COLOR_MAP` | 波色映射表 |
| `CONFIG.ELEMENT_MAP` | 五行映射表 |

### 三、全局固定三窗口

| 窗口名称 | 期数跨度 | 用途定位 |
|---------|---------|---------|
| 短期脉冲窗口 | 近5期 | 短期确认、快速信号捕捉 |
| 中期主趋势窗口 | 近10期 | 分池、打分、行情判定基准 |
| 长期稳定锚窗口 | 近15期 | 趋势校验、防骗线、风控兜底 |

### 四、数据输入与复用规则

1. 数据来源：仅读取项目原始历史开奖数据（期号 + 开奖号码）
2. 维度获取：调用 `DataQuery.getSpecial(item)` 自动获取每注号码对应的生肖、五行、波色、尾数、头数、区间
3. 窗口截取：自动从完整历史尾部截取固定三窗口（近5期、近10期、近15期）
4. 基础统计：各窗口内生肖开出频次、当前遗漏期数

### 五、生肖分池 & 打分算法（固定规则）

#### 5.1 统计基准
- 以近10期生肖遗漏为基准，计算遗漏均值 μ、标准差 σ

#### 5.2 热号池
- 条件：遗漏最小 TOP5 + 同遗漏值全部自动并入
- 固定得分：35 分

#### 5.3 温号池
- 条件：遗漏落在 `[μ-0.5σ , μ+0.5σ]`，排除热号池
- 评分公式：`得分 = 30 - 10 × (遗漏 - (μ-0.5σ)) / σ`
- 分值钳位：20~30 分
- 特殊规则：遗漏偏小非热号统一封顶 30 分

#### 5.4 冷号池
- 条件：剩余所有不在热/温池的生肖
- 基础分：10~18 分，遗漏越大分值越高
- 加分规则：近5期冷号开出 ≥3 期，冷号全体加分 +8，最高不超过 20 分

#### 5.5 分值硬性层级
- **热号 35 分 > 温号 20-30 分 > 冷号最高 20 分**
- 永远不允许倒挂

### 六、三窗口行情交叉判定规则

| 规则 | 说明 |
|------|------|
| 主次关系 | 以10期为主基调，5期做短期确认，15期做大趋势锚定 |
| 信号一致 | 10期与5期行情一致 → 正式锁定当前模式 |
| 信号背离 | 10期与5期背离 → 启用15期做最终锚定 |
| 散乱兜底 | 三期信号全部散乱 → 强制兜底默认走【正常轮动模式】 |
| 切换保护 | 行情模式切换后，强制锁定5期内禁止反复横跳 |

### 七、五大行情模式量化定义

| 模式 | 判定条件 |
|------|---------|
| **正常轮动** | 10期热号占比 25%-40%、温号开出最多；5期温号主导，冷热零星交替 |
| **强追热** | 10期热号占比 ≥40% 或 热号总数 > 温+冷总和；5期热号开出 ≥3 期 |
| **冷号接力** | 10期冷号占比偏高、有连续冷开出；5期冷号 ≥3 期且热号 ≤1 期 |
| **跳开乱序** | 5/10/15期三池分布散乱、无主力池、频繁跨池切换无规律 |
| **冷热交替均衡** | 5期内热/温/冷三池均有开出，无单池占比超 40%，无连续2期同池 |

### 八、多维度热度自动统计

1. **统计维度**：自动对 5/10/15 期分别统计生肖、五行、波色、尾数、头数、区间 开出频次
2. **热度分档**：各维度按频次降序排序，划分【热度高/中/冷】三档
3. **映射复用**：全部复用项目现有维度解析能力（`DataQuery` 模块），无需自建对照表

### 九、五大行情对应执行策略

| 模式 | 生肖策略 | 维度匹配 |
|------|---------|---------|
| 正常轮动 | 主推温号池，辅助少量热号，冷号仅兜底 | 中热五行、中热波色、中热尾数、主力区间 |
| 强追热 | 主推热号梯队、辅助温号次热，放弃冷号主推 | 各维度最热梯队 |
| 冷号接力 | 主推 10+15 期双窗口极值冷号 TOP3，温冷交界做辅助 | 极值超 μ+2σ 强制进防守池 |
| 跳开乱序 | 弱化生肖冷热池权重，完全依靠五行/波色/尾数/区间热度择优 | 多维热度择优 |
| 冷热交替均衡 | 热号只取TOP3分值下调为30，温号全保留，冷号取TOP2适度上浮；配比：热1+温3+冷2 或 热2+温3+冷1 | 中等热度维度全覆盖 |

### 十、内置风控保护机制

| 风控项 | 规则 |
|--------|------|
| 信号背离保护 | 三窗口信号背离，强制回归保守正常轮动 |
| 极值冷号标记 | 生肖遗漏 > μ+2σ 自动标记超级极值冷号，永久纳入防守备选池 |
| 层级锁定保护 | 严格锁定分值层级，任何场景不允许温号、冷号分数超过热号 |
| 冷却锁机制 | 行情模式设置冷却锁（5期），避免短期噪音频繁切换策略 |

### 十一、V3.2 优化规则层（8条永久开启）

> 在三窗口、分池、行情基础上，**追加过滤/评分/规避逻辑**，不修改原有函数名、接口、输出结构。

| 编号 | 名称 | 触发条件 | 执行动作 | 页面标签 |
|------|------|---------|---------|---------|
| 1 | 热号连开衰减 | 近5期≥4次 | 扣5分、降级备选、禁止精选首位 | `过耗` |
| 2 | 伪热三窗口过滤 | 5期≥1 + 10期前6 + 15期不掉队 | 扣3分、标记伪热 | `伪热` |
| 3 | 冷号临界反转 | 遗漏≥μ+1.4σ | 同类维度（五行/波色/尾数）回暖则+2~4分 | `临界` |
| 4 | 三窗口加权评分 | 全局自动执行 | 10期(50%) + 5期(30%) + 15期(20%) 归一化融入总分 | — |
| 5 | 精选3肖硬过滤 | 全局自动执行 | 三窗口共振 + 无连开透支 + 五行/波色分散 + 只取前三 | — |
| 6 | 连开透支规避（轻） | 近5期≥2次 | 移出精选 | — |
| 7 | 连开透支规避（重） | 近5期≥3次 | 彻底规避，禁止进入任何推荐池 | — |
| 8 | 上期重码零首选 | 上期开出生肖 | 直接终极规避，不进任何推荐池 | `重码` |

### 十二、V3.3 智能增强层（5条永久开启）

> 全部由程序**自动读取历史数据自行计算识别**，禁止手工写死任何生肖名单。

| 编号 | 名称 | 自动计算逻辑 | 执行动作 | 页面标签 |
|------|------|-------------|---------|---------|
| 1 | 年度弱势肖自动锁定 | 统计全年频次倒数+遗漏偏高 | 禁止进入精选3肖主推/备选，仅可放入终极防守池 | `年弱` |
| 2 | 同梯队热肖轮换互斥 | 全年频次+15期窗口聚类划分一线/二线/三线 | 精选3肖最多只入选1~2只一线热肖 | `_tier=1/2/3` |
| 3 | 阶段性热度透支静默 | 近15期密度超年均2倍且≥3次 | 扣8分、移出精选主推、进入短期静默 | `静默` |
| 4 | 热肖集体熄火预判跳开 | 一线热肖近5期连续2期集体不出 | 强制判定行情模式为 `chaos`（跳开乱序） | — |
| 5 | 近3期全域出号降权 | 自动抓取最新近3期所有已出生肖 | 统一减3分、降低精选排序权重 | `近3` |

### 十三、完整算法调用链路（强制执行顺序）

```
calc(historyData)
  ├─ _extractWindows          ← 截取5/10/15期
  ├─ _calcFreqStats            ← 频次+遗漏统计（μ/σ）
  ├─ _assignPools              ← 热温冷分池+基础打分
  ├─ _applyOptimizationRules   ← V3.2八条规则（衰减/伪热/反转/加权/透支/重码）
  ├─ _detectMarketMode         ← 三窗口交叉判定五大行情
  ├─ _calcDimensionHeat        ← 六维热度统计
  ├─ _calcAnnualWeakZodiacs    ← V3.3规则1：年度弱势肖
  ├─ _assignTierPools          ← V3.3规则2：梯队划分
  ├─ _getRecent3Zodiacs        ← V3.3规则5：近3期出号
  ├─ _detectHotShutdown        ← V3.3规则4：熄火预判
  ├─ _applyV33Rules            ← V3.3五条规则（弱势/梯队/静默/降权）
  ├─ _buildStrategy            ← 五档行情对应策略
  ├─ _applyRiskControl         ← 内置风控兜底
  └─ _buildSelected3           ← 精选3肖（全规则过滤）
```

### 十六、开发规范

- **业务层（business）**：所有数据拆解、统计、分池、打分、行情判定、策略逻辑统一写在 `business-gemini.js`
- **视图层（view）**：页面渲染、列表展示、排序全部写在 `view-gemini.js`
- **绝对禁区**：禁止在业务层写任何 DOM、渲染、视图操作代码
- **映射复用**：全程复用项目已有号码维度映射方法（`DataQuery` 模块），禁止重复自建对照表
- **代码规范**：模块化拆分、关键逻辑加注释，规则完整落地不简化、不私自改阈值
