# 追号计划算法优化方案

> 创建日期：2026-05-08
> 状态：待实施
> 目标：提升追号推荐命中率，增强算法鲁棒性

---

## 一、当前算法分析

### 1.1 核心逻辑

文件：[business-high-chase.js](business/business-high-chase.js) → `_recommendNext()`

```
推荐 = 固定窗口内平权频率 ≥ 阈值的生肖（按次数排序取前4）
```

### 1.2 流程

```
历史数据 → 市场判断(hot/normal/shock/cold) → 选窗口(27-32期)
       → 过滤最近N期出现过的生肖 → 统计频率 → 取前4个高频
```

### 1.3 弱点

| 弱点 | 说明 |
|------|------|
| 平权计数 | 窗口内第1期和第28期的权重一样，忽略时间衰减 |
| 单窗口 | 只用一个固定窗口，短期爆发和长期稳定无法区分 |
| 无视遗漏 | 不考虑生肖多久没出现（冷号回归信号） |
| 无视时序 | 不看生肖间的先后关系（牛后常跟龙？） |

## 二、优化方案

### 方案A：指数衰减加权（EMA-Frequency）⭐⭐⭐ 首选

#### 原理

每期出现不再是 `+1`，而是 `e^(-λ × age)`，距今越近权重越大。

```
加权频率 = Σ e^(-λ × age_i)

age_i  = 距今多少期
λ      = 衰减系数（默认 0.1）
```

#### 示例（λ=0.1, 窗口28期）

| 生肖 | 出现位置 | 传统频率 | EMA频率 |
|------|---------|:------:|:------:|
| 龙 | 3期前、8期前、20期前 | 3 | 1.325 |
| 兔 | 1期前、2期前、5期前 | 3 | 2.331 |

> 传统频率两者相同（都是3次），EMA频率正确识别出兔更"热"

#### 参数

| 参数 | 默认值 | 说明 | 调优方向 |
|------|:-----:|------|---------|
| `emaDecayLambda` | `0.10` | 衰减系数 | 增大 → 更看重近期；减小 → 更看重总量 |

#### 影响范围

- 修改：`_countZodiacFrequency()` 增加 EMA 模式
- 新增：`CHASE_CONSTANTS.EMA_DECAY_LAMBDA`
- 原有平权计算保留，通过开关切换

#### 实现量：约 5 行

---

### 方案B：多时间框架共识（Multi-Timeframe）⭐⭐⭐ 推荐组合

#### 原理

不用单一窗口，同时看短期(7期)、中期(15期)、长期(30期)，三个框架都高分的生肖才是真正稳定的热号。

```
综合分 = short_score × Ws + mid_score × Wm + long_score × Wl

默认权重: Ws=0.50, Wm=0.30, Wl=0.20
```

#### 示例

| 生肖 | 7期 | 15期 | 30期 | 综合分 | 判定 |
|------|:--:|:---:|:---:|:-----:|------|
| 龙 | 1 | 3 | 8 | 2.6 | 长期热但短期凉了 → 不推荐 |
| 牛 | 3 | 5 | 7 | 4.4 | 三个框架都热 → 推荐 ✅ |
| 蛇 | 4 | 4 | 3 | 3.7 | 短期爆发 → 可能是假热号 |

#### 参数

| 参数 | 默认值 | 说明 |
|------|:-----:|------|
| `multiWindowShort` | `7` | 短期窗口大小 |
| `multiWindowMid` | `15` | 中期窗口大小 |
| `multiWindowLong` | `30` | 长期窗口大小 |
| `multiWindowWeights` | `[0.5, 0.3, 0.2]` | 三窗口权重 |

#### 影响范围

- 修改：`_recommendNext()` 内跑三次统计，加权融合
- 新增：4个配置参数
- 保留单窗口模式，通过开关切换

#### 实现量：约 20 行

---

### 方案C：马尔可夫转移矩阵（Markov Transition）⭐⭐

#### 原理

不算"谁出现得多"，而算"当前生肖出现后，下一个最可能是什么"。

建立 12×12 转移概率矩阵：

```
P(A→B) = 生肖A后面出现生肖B的次数 / 生肖A出现的总次数
```

#### 条件

- 至少需要 **200期** 历史数据，否则矩阵太稀疏
- 转移概率可能随时间漂移，需定期重建

#### 参数

| 参数 | 默认值 | 说明 |
|------|:-----:|------|
| `markovMinHistory` | `200` | 最少需要的历史期数 |
| `markovRebuildInterval` | `50` | 每N期重建一次矩阵 |

#### 影响范围

- 新增：`_buildMarkovMatrix()` 构建转移矩阵
- 新增：`_predictByMarkov()` 基于当前生肖预测下一个
- 新增：3个配置参数

#### 实现量：约 40 行

### 方案D：综合评分引擎（Ensemble Scoring）⭐⭐⭐⭐ 终极方案

#### 原理

多个维度加权合并，各维度可独立开关：

```
最终得分 = 
    频率分(EMA)      × W_freq     ← 当前方案升级版
  + 遗漏分(冷号回归)  × W_miss     ← 新维度
  + 转移分(Markov)   × W_trans    ← 时序关联
  + 趋势分(动能方向)  × W_trend    ← 新维度

默认权重: [0.40, 0.25, 0.20, 0.15]
```

#### 四个维度详解

| 维度 | 计算方式 | 含义 |
|------|---------|------|
| 频率分 | EMA加权频率归一化到0-100 | 近期热不热 |
| 遗漏分 | `min(missCount/窗口长, 1) × 100` | 多久没出现（冷号回归概率） |
| 转移分 | 当前生肖→目标生肖的转移概率 × 100 | 时序关联 |
| 趋势分 | 最近5期频率 - 前5期频率，归一化 | 热度在上升还是下降 |

#### 参数

| 参数 | 默认值 |
|------|:-----:|
| `ensembleWeights` | `{ freq: 0.40, miss: 0.25, trans: 0.20, trend: 0.15 }` |
| `ensembleEnableFreq` | `true` |
| `ensembleEnableMiss` | `true` |
| `ensembleEnableTrans` | `true`（需200+期） |
| `ensembleEnableTrend` | `true` |

#### 影响范围

- 新增：`_scoreEnsemble()` 综合评分函数
- 修改：`_recommendNext()` 调用综合评分替代纯频率
- 新增：8个配置参数
- 保留原频率模式，通过开关回退

#### 实现量：约 80 行

## 四、回退机制

所有新算法均为**追加模式**，保留原有逻辑：

```javascript
// _recommendNext 顶层开关
const useAlgorithm = CHASE_CONSTANTS.ALGORITHM_MODE; 
// 'legacy' | 'ema' | 'multi' | 'ensemble'

if(useAlgorithm === 'legacy') {
  return _recommendLegacy(history);
}
if(useAlgorithm === 'ema') {
  return _recommendEMA(history);
}
// ...
```

一键切回原算法，零破坏。

## 五、效果验证方法

### 回测指标

| 指标 | 计算方式 | 目标值 |
|------|---------|:-----:|
| 命中率 | 命中计划数 / 总计划数 | > 60% |
| 平均追号期数 | 命中时平均用了多少期 | < 1.5 |
| 连续不中组数 | 连续全miss的计划组数 | < 2 |

### 对比方式

```
1. 记录当前算法最近30组计划的命中率作为 baseline
2. 实施阶段一（EMA频率）
3. 运行30组，对比命中率变化
4. 达标后继续阶段二
```
